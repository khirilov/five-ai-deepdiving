## 1. Як влаштований цей репозиторій

Дві половини, які навмисно живуть разом:

- **`src/`** — React-апка, яка і Є презентація (44 слайди, `pnpm dev` → localhost:5173,
  навігація ←/→/Space, диплінк `#/N`).

  Кожен код-приклад на слайді — цитата реального файлу звідси.

Головний прийом доповіді: **слайд → відкрив файл → запустив**. Ти ніколи не показуєш

Що ще прочитати в репо перед доповіддю (15 хв):

2. `.claude/README.md` — мапа демо англійською (можеш показати глядачам).
3. Файли, які цитують слайди: `grill-me/SKILL.md`, `deck-stats/SKILL.md`,
   `deck-review.md` (description), `slide-reviewer.md`, `tdd/SKILL.md` (frontmatter),

## 2. Теорія по блоках — пояснення «на пальцях»

> Цей розділ написаний так, ніби ти вперше чуєш ці слова. Спершу проста аналогія,

### 2.0 Карта на одній сторінці: 5 механізмів і коли який брати

Перш ніж занурюватися — ось уся доповідь у двох таблицях. Якщо запам'ятати тільки це,
ти вже зможеш користуватися Claude Code правильно.

**Спільна ідея.** Контекст (аркуш перед очима) — найдорожчий ресурс. Усі 5 механізмів
нижче — це способи дати Claude **більше**, не роздуваючи аркуш. Кожен закриває свій
тип задачі:

| Механізм      | Однією фразою                                  | Бери, коли…                                              |
| ------------- | ---------------------------------------------- | ------------------------------------------------------- |
| **CLAUDE.md** | факти, що завжди під рукою                      | це **завжди-істинний факт/конвенція** (стек, правила)   |
| **Скіл**      | покрокова інструкція на вимогу                  | це **повторювана процедура** (рев'ю, реліз, скаффолд)   |
| **Субагент**  | окремий працівник зі своїм аркушем              | треба **брудна автономна робота**, віддати лише висновок |
| **Хук**       | гарантія замість прохання                        | потрібно, щоб щось ставалося **завжди**, не на вибір моделі |
| **MCP**       | розетка в зовнішній світ                         | Claude має **читати/керувати зовнішньою системою** (браузер, GitHub) |

Мнемоніка: **факти → CLAUDE.md · процедури → скіли · автономна робота → субагенти ·
гарантії → хуки · зовнішній світ → MCP.** Плагін — це не шостий механізм, а **спосіб
запакувати й роздати** перші п'ять однією командою.

**Чесно про слайди — що критично, а що бонус** (відповідь на «чи всі слайди потрібні»):

- **Ядро доповіді (не ріж):** контекст/токени (2.1), скіли (2.2), субагенти (2.3),
  хуки (2.4), `/context` як живий доказ. Без цих п'яти решта не тримається.
- **Сильні підсилювачі:** ціна MCP (2.6), економія токенів (2.5) — це «вау-числа» й
  відповіді на найгостріші питання із залу.
- **Бонус, ріжеться першим за браку часу:** Figma, Perplexity, плагіни-наживо,
  learning-режим живцем. Їх лишаєш **розповіддю**, а демо вирізаєш (порядок різання —
  у §4). Тобто так, половину **демонстрацій** можна не показувати — але **теорію** з
  п'яти ядрових блоків лиши.

### 2.1 Контекст і токени — фундамент, без якого решта не має сенсу

**Аналогія.** Уяви Claude як дуже розумного працівника з ідеальною пам'яттю —
але тільки в межах **одного аркуша перед очима**. Все, що не на аркуші, для нього
не існує. Цей аркуш і є **контекст-вікно**.

**Що на цьому аркуші лежить (і за що ти платиш щоразу):**

1. **Системний промпт** — інструкція від Anthropic «ти Claude Code, ось як поводитися».
   Ти його не контролюєш.
2. **Визначення інструментів (tools)** — список того, що Claude вміє: читати файли,
   запускати команди, шукати. Кожен інструмент описаний текстом — це теж місце на аркуші.
3. **CLAUDE.md** і правила проєкту — твої постійні нагадування.
4. **Історія розмови** — усе, що ви вже сказали одне одному.
5. **Результати інструментів** — вміст файлів, які він прочитав, вивід команд, логи.

**Найважливіше для розуміння всієї доповіді:** модель **не пам'ятає** попередній
запит. Вона «без стану» (stateless). Тому **весь аркуш перечитується з нуля при
кожному твоєму повідомленні**. Не «дописується» — саме перечитується цілком.

**Що таке токен.** Це шматочок тексту (приблизно ¾ слова). Чим більше токенів на
аркуші — тим:

- **дорожче** (платиш за кожен токен у кожному запиті),
- **повільніше** (моделі треба «прочитати» більше перед відповіддю),
- і головне — **тупіше**. Це не метафора: коли важливе тоне в шумі, увага моделі
  розмазується, і вона частіше промахується. Чистий контекст = розумніші відповіді.

**Скільки це коштує в цифрах — чесно, з кешем.** Тут важливо не злякати неправдою.
Так, базовий «податок» (системний промпт + вбудовані інструменти) ≈ **20–25 тисяч
токенів**, і кожен MCP-сервер докидає свої описи: Chrome DevTools ≈ **18k**,
Playwright ≈ **13.7k**. Ці токени **фізично присутні в кожному запиті** (модель
stateless — увесь префікс пересилається щоразу). **АЛЕ це не значить, що ти платиш за
них повну ціну щоразу** — рятує **prompt caching** (детально в §2.5).

Як це працює на цінах **Opus 4.8** (офіційний прайс: input $5/1M, cache write 1.25× =
$6.25/1M, **cache read 0.1× = $0.50/1M**, output $25/1M):

- Префікс (база + MCP) — це незмінний початок кожного запиту, ідеальний кейс для кешу.
- **Перший** запит у сесії платить cache write один раз: ~33k токенів × $6.25/1M ≈
  **$0.21**.
- **Кожен наступний** запит з тим самим префіксом — це cache **read**: ~33k × $0.50/1M
  ≈ **$0.017**. Десь у **10 разів дешевше**.

Тобто «привіт» у вже прогрітій сесії коштує ≈ **1.7 цента**, а **не** «33k токенів на
повну ціну». Якщо хтось каже «кожне `привіт` палить 33k токенів» — це технічно про
кількість токенів правда, але про **гроші** — перебільшення приблизно в 10 разів.

**Два чесні застереження** (це і є реальний меседж, а не лякалка):

1. Кеш живе **5 хвилин** (або 1 год за ціну write 2×). Довша пауза → префікс випадає,
   наступний запит знову платить write. Тому черга швидких запитів дешевша за рідкісні.
2. **Кешується лише незмінний префікс.** Історія чату росте: кожна твоя нова відповідь
   і кожен вивід тула **дописуються** в хвіст, який кешу ще нема — його доводиться
   писати (1.25×), і далі він теж читається (0.1×). **Брудний довгий чат = більший
   хвіст, який щоразу читається, навіть по 0.1×.** Ось **справжня** причина, чому
   `/clear` між задачами економить: ти скидаєш хвіст до нуля.

**Звідси — головна теза всієї доповіді.** Скіли, субагенти й хуки — це три різні
способи дати Claude **більше знань і можливостей, НЕ роздуваючи аркуш**. Кожен робить
це по-своєму, і саме тому їх три, а не один. Далі — по черзі, що кожен дає і коли
який брати.

> 💡 Одна команда, яку треба запам'ятати з усієї доповіді: **`/context`**. Вона
> показує, чим саме зайнятий твій аркуш, по категоріях. Будь-яку оптимізацію починай
> з неї — не «на око».

### 2.2 Скіли — багаторазова інструкція «дістається з полиці на вимогу»

**Аналогія.** Уяви теку з інструкціями на полиці: «як ми робимо реліз», «як писати
опис MR», «як проводити код-рев'ю». Ти не тримаєш усі ці папери розгорнутими перед
очима постійно — ти береш потрібну, коли вона потрібна, і кладеш назад. Скіл — це
рівно така інструкція: лежить згорнута, розгортається тільки коли треба.

**Що це технічно.** Скіл — це **папка** з файлом `SKILL.md` усередині. Сам файл —
звичайний markdown із двох частин:

```text
.claude/skills/grill-me/
└── SKILL.md
        ├── frontmatter  ← «паспорт» скіла між --- --- (name, description, налаштування)
        └── тіло          ← сама інструкція звичайним текстом: «зроби це, потім те»
```

Скіл може жити у двох місцях:

- `.claude/skills/` у репо → **проєктний**, комітиться в git, доступний усій команді;
- `~/.claude/skills/` → **особистий**, працює в усіх твоїх проєктах.

**Головна суперсила — прогресивне розкриття (progressive disclosure).** Це і є
причина, чому скіли кращі за «запхати все в CLAUDE.md». Працює в три рівні:

1. **Завжди в контексті висить лише `description`** (~100 токенів на скіл) — короткий
   рядок «що це й коли застосовувати». Тіло скіла **не завантажене**.
2. **Коли скіл активується** — Claude дочитує повне тіло. Тільки в цей момент, тільки
   цей скіл.
3. **Допоміжні файли поруч** (наприклад `tdd/tests.md`, `tdd/mocking.md`) — ще глибший
   рівень: читаються, лише коли Claude дійде до відповідного кроку всередині скіла.

Порівняй із CLAUDE.md: він вантажиться **весь і завжди**, у кожному запиті. Тому
правило: **факти й конвенції → CLAUDE.md; покрокові процедури → скіли.** Скіл майже
безкоштовний, поки ти ним не користуєшся.

**Коли взагалі писати скіл (а коли не треба).** Сигнали, що пора:

- ти **втретє** копіюєш ту саму інструкцію в чат;
- шматок CLAUDE.md перетворився з «факту» на «покрокову інструкцію»;
- задача має повторювану форму: рев'ю, реліз, скаффолд, міграція.

**Не** роби скіл, якщо: це разова дія (просто попроси), це завжди-істинний факт
(→ CLAUDE.md), потрібен зовнішній інструмент (→ MCP), або потрібна **гарантія**, а не
прохання (→ хук). Коротка мнемоніка: **факти → CLAUDE.md · процедури → скіли ·
гарантії → хуки · зовнішній світ → MCP.**

**Хто запускає скіл — і чому це робить description найважливішим полем.** Два шляхи:

- **Ти** — явно набираєш `/grill-me`, `/feature-forge`.
- **Claude сам** — читає `description` усіх скілів і вирішує, чи якийсь підходить до
  твого запиту. Якщо description написаний погано — Claude просто не здогадається
  активувати скіл, і той ніколи не спрацює.

Тому description пишемо **«наполегливо», за формулою: _що робить_ + _"Use when: ..."
з конкретними тригерами_** (фразами й прикладами запитів). Важливий нюанс: Claude
радше **недо**використовує скіли, ніж зловживає ними — тому опис має буквально
«продавати» скіл. Технічний ліміт: `description` + `when_to_use` разом ≤ 1536
символів у листингу.

Приклад доброго description (наш `deck-review`):

```yaml
description: "Review the whole slide deck against repo conventions by fanning out
  one slide-reviewer subagent per section. Use when: checking the deck before the
  talk, auditing slides for conventions/overflow/quote drift, or mentions 'deck review'."
```

Бачиш структуру: спочатку **що робить**, потім **"Use when:"** з трьома конкретними
тригерами. Поганий варіант був би просто `description: "Reviews the deck"` — Claude
ніколи не зрозуміє, коли це доречно.

**Команди = скіли (просто старіший формат).** Файл `.claude/commands/x.md` і папка
`.claude/skills/x/SKILL.md` **обидва** створюють `/x` і працюють однаково. Commands —
старіший формат (один файл, без папки й допоміжних файлів). Skills — новіший (папка +
допоміжні файли + повний набір налаштувань у frontmatter). Для нового — пиши skills.

**Frontmatter — «паспорт» скіла. Кожне поле і навіщо воно (це й питають із залу):**

| Поле                             | Що дає (людською мовою)                                                     | Наш приклад             |
| -------------------------------- | -------------------------------------------------------------------------- | ----------------------- |
| `name`                           | ім'я в листингу (за замовчуванням = ім'я папки)                            | усі                     |
| `description` / `when_to_use`    | **головне поле**: за ним Claude вирішує, чи активувати скіл сам             | усі                     |
| `argument-hint`                  | підказка в автокомпліті: `/feature-forge [feature name]`                    | feature-forge           |
| `arguments`                      | іменовані аргументи → в тілі стають `$name`                                 | (показати на слайді)    |
| `disable-model-invocation: true` | Claude **НЕ може** запустити сам — тільки ти руками. Для дій із наслідками: деплої, релізи, міграції | **feature-forge(-solo)** |
| `user-invocable: false`          | сховати з меню `/` — це фонові знання, не дія («як влаштований наш білінг») | слайд-приклад           |
| `allowed-tools`                  | тули, дозволені без запиту, поки скіл активний (напр. `Read, Glob, Grep`)   | deck-stats              |
| `disallowed-tools`               | тули **заборонені**, поки скіл активний                                     | —                       |
| `model` / `effort`               | перемкнути модель або зусилля на час роботи скіла (haiku для рутини)        | —                       |
| `context: fork` + `agent`        | тіло скіла виконується в **окремому** контексті-форку (див. нижче)          | **deck-stats**          |
| `paths`                          | скіл сам активується тільки для файлів за глобом (`"src/**/*.tsx"`)         | —                       |
| `hooks`                          | хуки, що діють тільки поки скіл активний                                    | **tdd**                 |

**Два поля, на яких варто зупинитися окремо, бо вони рятують від проблем:**

- `disable-model-invocation: true` — без нього агресивний description («Use when:
  implementing a feature…») змусив би Claude запускати 5-фазний процес на будь-яке
  «зроби фічу». Правило: **дія з великими наслідками = тільки ручний запуск.**
- `context: fork` — тіло скіла виконується в **окремому контекст-вікні** (форку),
  і в твою основну розмову повертається **лише результат**, а не весь процес. Наш
  `/deck-stats` так і працює: читає всі файли слайдів у форку, а тобі віддає тільки
  табличку. Покажи `/context` до і після — основне вікно не виросло.

**Підстановки в тілі — як скіл стає шаблоном.** У тілі скіла можна писати плейсхолдери,
які підставляються в момент запуску:

| Плейсхолдер           | Що підставляється                                                |
| --------------------- | ---------------------------------------------------------------- |
| `$ARGUMENTS`          | усе, що ти написав після `/name`                                 |
| `$0`, `$1`, …         | окремий аргумент за позицією                                     |
| `$issue` (будь-яке ім'я) | іменований аргумент із `arguments: [issue, ...]`              |
| `${CLAUDE_SKILL_DIR}` | папка самого скіла — щоб посилатися на скрипти поруч              |
| `` !`git status` ``   | **команда виконується**, її вивід вставляється в тіло (живий контекст) |

Останнє — потужне: `` !`git status --short` `` означає «у момент запуску виконай цю
команду й встав результат сюди», тож скіл бачить актуальний стан, а не вигадує його.

**9 правил, як писати хороші скіли (це і є відповідь на «як правильно складати»).**
Кожне: що написати → яку поведінку купуєш:

1. **`description` = «що робить + Use when: тригери»** → Claude реально сам активує
   скіл. Розмитий опис = скіл, що ніколи не спрацьовує.
2. **«Якщо відповідь є в кодовій базі — дослідь код, не питай»** (один рядок у тілі) →
   прибирає ~80% дурних уточнювальних питань.
3. **Явні фази + «No proceeding without approval»** → модель не «пролітає» одразу в
   код на першому ж повідомленні. Текст скіла = поведінкові рейки.
4. **`disable-model-invocation: true` для деплоїв/релізів** → дії з наслідками
   лишаються ручними, не самозапускаються.
5. **Persist артефактів у файли** (спека → `docs/specs/...`) → довгий процес переживає
   `/compact` і навіть нову сесію. Контекст смертний, файли — ні.
6. **Композиція по імені** (grill-me → design → tdd) → маленькі скіли
   перевикористовуються окремо; оркестратор тримає лише процес.
7. **Брифи субагентам: різні обмеження + жорсткий формат звіту + ліміт рядків** →
   паралельні агенти повертають порівнюване, а не три есеї (деталі — §2.3).
8. **Секція Enforcement Rules** — правила, які скіл **цитує тобі ж**, коли ти
   намагаєшся зрізати кут (на демо спробуй сказати «давай одразу код» — він відмовить).
9. **`TodoWrite` на старті** → і ти, і глядач бачите прогрес фаз у тудушках.

**Головна думка про тіло скіла:** це не документація, а **поведінкові рейки**. Пиши
саме те речення, яке прибирає помилку, яку ти втомився виправляти вручну.

### 2.3 Субагенти — окремий працівник зі своїм власним аркушем

**Аналогія.** Повернись до образу працівника з аркушем. Субагент — це **інший**
працівник, якому ти даєш окреме завдання, **свій чистий аркуш** і кажеш: «зроби це
й принеси мені короткий висновок». Він іде в окрему кімнату, перериває там 50 файлів,
списує цілий блокнот чернеток — а тобі приносить **один абзац підсумку**. Усі його
чернетки лишаються в його кімнаті й твій аркуш не засмічують.

**Що це технічно.** Субагент — це файл `.claude/agents/<name>.md`. У нього:

- **власне контекст-вікно** (окремий аркуш) — головна причина існування;
- **власний системний промпт** (своя «роль»);
- **обмежений набір інструментів** (можна дати тільки читання);
- життєвий цикл простий: **отримав промпт → попрацював сам → повернув ОДИН звіт.**

**Навіщо це потрібно — конкретне число для сцени.** Дослідження по 50 файлах легко
з'їдає ≈ **40k токенів** (увесь вміст файлів, який Claude переглянув). Підсумок із
цього дослідження ≈ **500 токенів**. Якщо робити це в головній розмові — ті 40k
**осядуть у твоєму контексті назавжди** і дорожчатимуть кожен наступний запит. Якщо
це робить субагент — у твою розмову повертаються тільки ті 500 токенів звіту. Ось і
вся магія: **брудну роботу робить хтось зі своїм контекстом, тобі віддає чистий
результат.**

**Поля frontmatter субагента і навіщо кожне:**

| Поле                  | Що дає                                                                       |
| --------------------- | --------------------------------------------------------------------------- |
| `name`, `description` | обов'язкові; за `description` Claude вирішує делегувати завдання сам         |
| `tools` / `disallowedTools` | які інструменти доступні. **Read-only агент = безпечний агент** — не зламає нічого |
| `model`               | дешева модель під рутину (`haiku`) — наш `slide-reviewer` саме так           |
| `maxTurns`            | стоп-кран: максимум кроків, щоб агент не «закопався» назавжди                |
| `permissionMode`      | режим дозволів усередині агента                                             |
| `skills`              | скіли, що **повністю влиті** в контекст агента на старті (не на вимогу, а одразу) |
| `memory`              | агент пише собі нотатки між сесіями (user / project / local)                |
| `isolation: worktree` | агент працює в **окремій копії репо** — не чіпає твої файли                 |
| `background`          | агент працює у фоні, ти продовжуєш своє                                     |

**Три вбудовані субагенти** (їх не треба писати, вони вже є):

- **Explore** — read-only пошук по кодовій базі. Замість того щоб самому читати 30
  файлів і засмічувати контекст — кажеш Explore «знайди, де визначено X», він шукає у
  своєму вікні, повертає коротку відповідь.
- **Plan** — архітектор: складає план реалізації.
- **general-purpose** — універсальний на будь-яку багатокрокову задачу.

**Найважливіше питання: скіл чи субагент? Відповідь — інтерактивність.**

Запам'ятай це одне правило, бо саме воно пояснює всю структуру нашого репо:
**субагент НЕ може говорити з користувачем посеред роботи.** Він бере промпт і
повертає звіт — жодних питань тобі по ходу.

Звідси випливає:

- Усе, що має **інтерв'ювати тебе** (ставити питання, чекати відповіді) — мусить бути
  **скілом/командою в головній розмові**: `grill-me`, `mr-reviewer`, `feature-forge`,
  `deck-review`.
- Усе, що **автономне й read-only** (отримав завдання — мовчки зробив — віддав звіт) —
  робимо **субагентом**: `slide-reviewer`, `design-explorer`, mantine-воркери.

Коротко: **скіл = ЯК робити** (інструкція в поточній розмові), **субагент = ХТО
робить** (окремий контекст). І вони комбінуються: `context: fork` запускає скіл у
субагенті; поле `skills` вливає скіли в агента.

**Патерн fan-out / fan-in (розгалуження → злиття) — як паралелять роботу.** Уяви:
оркестратор у головній розмові розсилає **N однакових завдань N субагентам одночасно**,
вони працюють паралельно (кожен у своєму вікні), а потім (за бажанням) ще один агент-
**reconciler** зливає їхні звіти в один результат. Це дає швидкість і незалежність:
агенти не бачать один одного, тому не «якоряться» на чужій ідеї.

Два наші приклади:

- **`/deck-review`** — живий, у цьому репо: на кожну секцію слайдів спавниться окремий
  `slide-reviewer` (на дешевому haiku), усі паралельно → зведений звіт.
- **`/mantine-css-auditor`** — з робочого проєкту (Mantine тут немає, тому це
  розповідь): оркестратор → N read-only воркерів → reconciler за правилом «бери лише
  те, з чим згодні всі».

### 2.4 Хуки — гарантія замість прохання

**Головна теза (це треба сказати на сцені дослівно).** Скіл чи рядок у CLAUDE.md —
це **прохання**: «будь ласка, запускай тести після правок». Claude може забути, бо це
його рішення. **Хук — це гарантія.** Це твій код, який Claude Code виконує **сам, без
участі моделі**, у потрібний момент. Хук не забуває ніколи, бо забувати нема кому —
це не рішення, це автоматика.

**Аналогія.** Скіл — це інструкція на стіні «мий руки перед їжею». Хук — це кран, що
**сам** вмикається, коли ти підходиш. Перше можна проігнорувати, друге — ні.

**Як це влаштовано — по кроках.** У файлі `settings.json` ти описуєш ланцюжок:

```text
подія  →  matcher (на який інструмент реагувати)  →  що зробити
```

- **подія** — момент, коли хук спрацьовує (список нижче);
- **matcher** — на який інструмент реагувати: ім'я (`Bash`), кілька через `|`
  (`Edit|Write`), або regex (`mcp__.*` — будь-який MCP-тул);
- **дія** буває кількох типів: `command` (запустити скрипт), `prompt` (спитати модель
  так/ні), `agent` (спавнити субагента-перевіряльника), `http`, `mcp_tool`.

**Контракт command-хука — як скрипт «розмовляє» з Claude Code.** Коли хук спрацьовує,
твоєму скрипту на **stdin** приходить JSON із деталями (id сесії, що за інструмент,
які аргументи). Скрипт відповідає **кодом виходу**:

- **exit 0** — усе гаразд, пропустити (у stdout можна повернути JSON із рішенням);
- **exit 2** — **заблокувати** дію, а текст зі stderr повертається Claude як фідбек —
  і він піде це виправляти.

Окремо PreToolUse-хук може повернути явне рішення: `permissionDecision: allow | deny |
ask`.

**Події (їх 30+, ось головні для розповіді).** Назва події = момент у житті сесії:

- **SessionStart** — старт сесії (так працює learning-плагін: докидає інструкції);
- **UserPromptSubmit** — щойно ти надіслав повідомлення;
- **PreToolUse** — **перед** викликом інструмента (тут можна **заблокувати**);
- **PostToolUse** — **після** виклику (тут запускаємо typecheck після правки);
- **PermissionRequest** — коли потрібен дозвіл;
- **Stop** — Claude закінчив відповідь (у нас тут звуковий сигнал);
- **SubagentStart / SubagentStop** — навколо роботи субагента;
- **PreCompact** — перед стисненням історії;
- **SessionEnd** — кінець сесії.

Подивитися, які хуки зараз активні й звідки взялися: команда **`/hooks`** (read-only
список із джерелами — звідки кожен хук прийшов).

**Рівні (де може жити хук, від ширшого до вужчого):**

- **глобальні** — `~/.claude/settings.json`, діють у всіх твоїх проєктах;
- **проєктні** — `.claude/settings.json`, комітяться в git → це правила команди;
- **локальні** — `settings.local.json`, тільки твоя машина, не в git;
- **скоуповані** — прямо у frontmatter скіла чи агента: хук діє **тільки поки той
  активний** (наш `tdd` так і робить — хук живе лише під час TDD-сесії).

**Наші три проєктні хуки (`.claude/settings.json`) — що показати пальцем:**

1. **PostToolUse на `Edit|Write` → typecheck** (асинхронно, статус у статус-барі):
   щойно Claude щось редагує — автоматично летить `pnpm typecheck`. Гарантія, що
   правки не ламають типи, без жодного «не забудь перевірити».
2. **PreToolUse на `Bash` → `block-git-push.mjs`**: перехоплює `git push` і робить
   **deny**. Це наш «запобіжник» — на демо проси Claude запушити, і хук відмовить.
3. **Stop → звуковий сигнал**: Claude закінчив — ти чуєш звук, не треба дивитися в
   екран.

> ⚠️ Важлива деталь для демо: **хуки читаються на старті сесії.** Якщо ти правиш
> `settings.json` — зміни підхопляться **тільки після рестарту сесії**. Не дивуйся на
> сцені, що свіжий хук не спрацював одразу.

### 2.5 Токени: практики економії

**Нагадування, навіщо це все.** Менше токенів = дешевше, швидше і **розумніше** (увага
моделі не розмазана). Усе нижче — конкретні важелі, від найпростіших до тонших. Перший
крок завжди один: подивитись `/context` і побачити винуватця, а не оптимізувати наосліп.

1. Діагностика перш за все: `/context` — розклад вікна по категоріях.
2. `/clear` між незалежними задачами — найнедооціненіша команда.
3. `/compact` свідомо на довгих сесіях (авто-microcompact теж є, але керований кращий).
4. Дешеві моделі під рутину: `model: haiku` у субагентах (+ `maxTurns`).
   На рівні сесії — `/model`: `default` (Opus → Sonnet після ~50% usage),
   **`opusplan`** (Opus думає в Plan Mode → Sonnet пише код — розумна модель лише
   там, де платить), або `sonnet`/`haiku` напряму. Слайд про це — Tokens (#29).
5. `effort` — глибина «думання» і витрати: `low` (субагенти/рутина) · `medium`
   (прості, cost-sensitive) · `high` (дефолт Claude Code, більшість роботи) ·
   `xhigh` (кодинг/агентика) · `max` (correctness > cost, може перемудрити).
   Ставиться у frontmatter скіла/агента (`effort: low`) або глобально.
6. Вимкнути невикористовувані MCP (найбільший прихований податок).
7. **Prompt caching** (слайд #31) — найбільший недооцінений важіль. Кеш = prefix-match:
   1 змінений байт у префіксі → все після нього некешоване (×10 дорожче). Рендер-порядок
   `tools → system → messages`, тому волатильне (дата, id) — в кінець. Що ламає кеш:
   правка CLAUDE.md/скіла посеред сесії, додавання/зняття MCP, зміна моделі. Це і є
   механізм «роздутий CLAUDE.md = податок»: він сидить на початку кешованого префікса.
8. Процедури з CLAUDE.md → у скіли (description висить, тіло — на вимогу).
9. **caveman** (плагін): ріже балачки в output, −65% у середньому (розкид 22–87%),
   код ідентичний, thinking не чіпає. Режими lite/full/ultra. Чесна примітка
   на сцені: це економія на видимих відповідях, не на «мисленні».
10. Моніторинг: `/usage`, зовнішній `ccusage`.

### 2.6 MCP — «USB-порт», через який Claude дотягується до зовнішнього світу

**Аналогія.** Скіли/агенти/хуки дають Claude **знання й процеси**. Але сам по собі він
не вміє відкрити браузер, прочитати твій GitHub чи зазирнути в базу даних — це все
**поза** його аркушем. MCP — це **універсальна розетка**: підключаєш «пристрій»
(браузер, GitHub, Figma), і Claude отримує набір нових кнопок, якими може це
керувати. Один стандарт на всі пристрої — як USB.

**Що це технічно.** MCP (Model Context Protocol) — **відкритий протокол**. Працює так:

- є **MCP-сервер** — окрема програмка, що «оголошує» список своїх інструментів
  (наприклад, браузерний сервер оголошує «зроби скріншот», «клікни», «прочитай
  консоль»);
- Claude бачить ці інструменти під іменами `mcp__<сервер>__<тул>` і викликає їх так
  само, як вбудовані Read чи Bash.

**Транспорт** (як Claude спілкується із сервером) буває двох видів:

- **stdio** — сервер це локальний процес на твоїй машині (запускається через `npx`);
- **http** — сервер віддалений, десь у мережі (тоді часто потрібен OAuth-логін).

**Скоупи — куди записується конфіг сервера (від вузького до широкого):**

- **local** — тільки ти, тільки цей проєкт (`~/.claude.json`). Сюди кладемо сервери з
  секретними ключами — щоб ключ не потрапив у git.
- **project** — `.mcp.json` у корені репо, **комітиться** → вся команда отримує ті самі
  інструменти, склонувавши репо.
- **user** — усі твої проєкти.

При збігу імен виграє вужчий: **local > project > user.**

**Безпека.** Проєктні сервери (з `.mcp.json`) Claude Code просить **підтвердити при
першому запуску** — це захист від того, що хтось підсунув у репо свій конфіг. Сам цей
діалог підтвердження — теж матеріал для демо (саме так команда ділиться інструментами
через git).

**Ціна — найчастіше питання із залу, тож розжовую по-чесному.** Число «DevTools ≈ 18k
токенів» — це розмір **описів (схем) інструментів** сервера, а **не** «спалюється
18k при кожному кліку». Точна механіка в три кроки:

1. **Схеми лежать у контексті й, оскільки модель stateless, відправляються з кожним
   запитом.** Саме цього бояться. АЛЕ — через prompt caching цей префікс кешується, і
   повторне читання ≈ **в 10 разів дешевше**. Тобто платиш переважно **місцем на
   аркуші** та **увагою** моделі, а не повною ціною токенів щоразу.
2. **Новіші версії Claude Code взагалі ВІДКЛАДАЮТЬ завантаження схем** (tool search):
   поки інструмент не потрібен — у контексті лише його **ім'я**, а повна схема
   підтягується тільки при першому використанні. Тобто стоячий податок ще менший.
3. **Сам виклик інструмента коштує окремо:** аргументи виклику + його результат осідають
   в історії. Снепшот сторінки від Playwright легко кілька тисяч токенів — і ось **це
   вже не кешується**.

Висновок простий і незмінний: **реальну вагу дивись у `/context`, зайве вимикай.**
Default — вимкнено: вмикай сервер під задачу, вимикай після.

> ⚠️ **Безпека-пастка (обов'язково проговорити).** Інструменти, що читають **зовнішній
> контент** (GitHub issues/PR, чужі сторінки) — це вектор **prompt injection**: у тому
> тексті може бути захована інструкція для моделі. Користуйся такими тулами лише на
> об'єктах, яким довіряєш.

**Наш `.mcp.json`:** chrome-devtools + playwright + context7 (без секретів; ключі —
тільки в local scope).

**Різниця DevTools vs Playwright (топ-питання із залу):** DevTools = СПОСТЕРІГАТИ
(консоль, network, перф-трейси, Lighthouse; CDP, тільки Chromium; ~26 тулів ≈ 18k
токенів) — питання «чому повільно/чому помилки». Playwright = ДІЯТИ (драйв юзер-флоу
через accessibility-снепшоти; крос-браузер; ~21 тул ≈ 13.7k) — питання «чи працює
флоу e2e». Не конкуренти. Обидва разом постійно = ~32k податку — вмикай під задачу.

**Figma Dev Mode** (нема доступу — слайд): desktop app + Dev/Full seat (Professional+),
Preferences → Enable Dev Mode MCP Server → `http://127.0.0.1:3845/mcp`; Claude читає
реальну структуру макета (auto-layout, варіанти, токени), не скріншот.

**GitHub:** MCP **не потрібен** — `gh` CLI (вже стоїть) покриває все. Офіційний
`api.githubcopilot.com/mcp` через `claude mcp add` дав `Incompatible auth server: does
not support dynamic client registration` (OAuth DCR не підтримується), тож пішли через
`gh`. Зв'язка з нашим `/github-pr-summary`: згенерував опис → `gh pr create --draft` без
виходу з термінала. Це й є живий приклад тези «не вмикай MCP, якщо CLI вистачає». (Якщо
MCP таки треба — PAT у заголовку Authorization, токен у local config.)

**Perplexity (Sonar):** пошук, що повертає готову відповідь з цитатами замість тонн
HTML → економія токенів. Потрібен платний API-ключ → local scope.

### 2.7 Плагіни і Learning mode

**Аналогія.** Досі ми писали скіли, агентів і хуки **руками** в `.claude/`. Плагін —
це коли хтось уже написав корисний набір таких речей, **запакував** і виклав, а ти
ставиш **однією командою**. Як застосунок зі стора: всередині той самий код, але тобі
не треба його писати — просто «встановити».

**Плагін** = **формат дистрибуції** (упаковки). Один пакет може містити скіли +
агентів + хуки + MCP-сервери **разом**. Ставиться глобально (`~/.claude/plugins/`) і
працює в усіх проєктах. Ключова думка для сцени: **усередині плагіна — ті самі
скіли/агенти/хуки, що ми пишемо руками; плагін просто зручна обгортка.**

**Маркетплейс** = git-репо зі списком пакетів (як «стор»). Команди: `claude plugin marketplace add <owner/repo>`,
`claude plugin install <plugin>@<marketplace>`, меню `/plugin`.
Каталог: claude.com/plugins (офіційний маркетплейс підключений за замовчуванням).

Імена для сцени: **frontend-design** (Anthropic, ~830k встановлень — виразний UI
замість «AI-слопу»), **caveman** (токени), **typescript-lsp** (діагностика типів як
в IDE), **context7** (той самий MCP, але плагіном), **superpowers** (~750k — приклад
масштабу екосистеми).

**Learning mode** (бонус-слайд): НЕ MCP — режим спілкування (output style), тепер і
офіційний плагін `learning-output-style`. Під капотом це просто інструкції, додані до
системного промпта (плагін робить це SessionStart-хуком) — ще один доказ тези «все
розширення Claude = текст у правильному місці».

- **Learning**: Claude пише обв'язку сам, але на змістовних рішеннях (бізнес-логіка з
  кількома валідними підходами, обробка помилок, вибір алгоритму) зупиняється, готує
  місце і лишає `TODO(human)` — «напиши ці 5–10 рядків сам», пояснивши трейдофи.
  Очевидне/boilerplate/CRUD він все одно пише сам — ти працюєш тільки над тим, що вчить.
- **Explanatory**: Claude працює як зазвичай, але кожне неочевидне рішення супроводжує
  блоком `★ Insight` з 2–3 пунктами «чому так».

**Як виглядає сесія в Learning:** просиш фічу → Claude скаффолдить і каже «у файлі X
є TODO(human), ось контекст і два можливі підходи» → ти пишеш свій шматок у файлі →
кажеш «готово» → Claude рев'юїть твій код і рухається далі.

**Вмикання (актуально для CLI ≥ 2.1.91):** `/config` → **Output style** → Learning /
Explanatory (є ще Proactive). Вибір зберігається в `.claude/settings.local.json` як
`"outputStyle": "Learning"` — можна виставити й руками. ⚠️ Системний промпт читається
раз на старті сесії, тому стиль вмикається **після `/clear` або нової сесії** — на
демо переключи заздалегідь! Стара команда `/output-style` видалена у v2.1.91 — не
показуй її. Альтернатива для дистрибуції на команду: плагін
`learning-output-style@claude-plugins-official` (він уже встановлений на цій машині).
**Вимикання:** `/config` → Output style → Default — інструкції режиму додаються в
системний промпт кожної сесії, тож поза навчанням це зайві токени.

Цикл «вивчи будь-що»: маленький проєкт → learning mode → ти заповнюєш пропуски →
`/grill-me` по написаному → повторити.

## 5. Ймовірні питання із залу (і відповіді)

- **«Чим скіл відрізняється від CLAUDE.md?»** — CLAUDE.md вантажиться весь і завжди
  (факти, конвенції). Скіл — на вимогу (процедури). Правило: якщо шматок CLAUDE.md
  став покроковою інструкцією — виноси в скіл.
- **«Скіл vs субагент?»** — скіл = ЯК робити (інструкція в поточній розмові),
  субагент = ХТО робить (окремий контекст). Комбінуються: `context: fork` запускає
  скіл у субагенті; поле `skills` вливає скіли в агента.
- **«Чи може субагент спавнити субагентів / спитати мене?»** — ні. Оркеструє головна
  розмова. Тому інтерактивні речі — команди, автономні — агенти.
- **«Хуки — це ж виконання довільного коду?»** — так, тому: тільки свої/перевірені,
  `/hooks` показує що звідки прийшло, проєктні хуки проходять код-рев'ю як код.
- **«Скільки «коштує» MCP-сервер?»** — схеми тулів сидять у префіксі: DevTools ≈ 18k,
  Playwright ≈ 13.7k токенів. Вони пересилаються щоразу, **але через prompt caching це
  cache read ≈ 0.1× ціни** (Opus 4.8: $0.50 замість $5 за 1M) — не повна ціна щоразу.
  Реальна вага видно в `/context`; найдорожче — не префікс, а роздутий хвіст історії.
- **«А якщо два скіли підходять?»** — Claude обирає за description; завжди можна явно
  `/name`. Конфлікти лікуються чіткішими "Use when".
- **«Чи їде наш код кудись через MCP?»** — MCP-сервер ти обираєш сам (локальний процес
  або твій/вендорський endpoint). У модель потрапляє те, що тул повернув у розмову —
  як і будь-який інший результат інструмента.
- **«Як поділитися цим з командою?»** — закомітити `.claude/` і `.mcp.json`. Особисте —
  у `~/.claude/`. Третій рівень — зробити свій плагін-маркетплейс.
- **«Хук може зупинити Claude взагалі?»** — так: PreToolUse deny блокує дію,
  `continue: false` зупиняє хід повністю, exit 2 повертає фідбек на виправлення.
- **«Це працює не тільки з Claude?»** — скіли: формат Agent Skills — відкритий стандарт
  (agentskills.io), MCP — відкритий протокол, який підтримують й інші клієнти.
  Хуки/агенти в цьому вигляді — специфіка Claude Code.

## 7. Демо feature-forge: фічі, сценарій по фазах, уроки для слухачів

### Які фічі тригерити

| Коли                     | Фіча                                                                                      | Що набирати                                                                                                                   |
| ------------------------ | ----------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **Репетиція (сьогодні)** | Section jump: клік на назву секції у футері → список секцій → перехід на її перший слайд  | `/feature-forge-solo section jump menu in the deck footer` → `/clear` → `/feature-forge section jump menu in the deck footer` |
| **Демо (на доповіді)**   | Talk timer: таймер у футері — скільки йде доповідь, ціль 40 хв, підсвітка при перевищенні | `/feature-forge talk timer in the deck footer with a 40-minute target` (через **Tab**!)                                       |

**Чому саме такі фічі:** малі (хук + шматок футера), результат видно на екрані одразу,
користь справжня (таймер реально потрібен тобі на доповіді — гарна іронія для залу),
і головне — **простір дизайну не вироджений**: «мінімальна поверхня» (хук
`useTalkTimer` що повертає рядок), «гнучкість» (компонент `<TalkTimer>` з пропсами
target/warn/format), «оптимізація під найчастіший кейс» (конфіг у деці + нуль API)
дадуть реально різні інтерфейси. На тривіальній фічі субагенти повернули б три
однакові відповіді — і демо здулося б.

Репетиційну фічу (section jump) після прогону або лишай і закоміть (вона корисна),
або відкоти: `git checkout -- src` (тому git init — ДО репетиції).

### Що відбувається по фазах і що ГОВОРИТИ

**Phase 1 — DISCOVER (~3 хв на демо).** Скіл змушує Claude працювати продактом:
структуровані питання через AskUserQuestion, кожне з рекомендованим дефолтом.
Очікувані питання для таймера: стартувати автоматично з першого слайда чи по клавіші?
формат (mm:ss)? що при перевищенні 40 хв (колір/блимання)? де саме у футері?
_Говорити:_ «зверни увагу — він не питає того, що може знайти в коді: це правило
прямо в тілі скіла». На демо обирай Recommended-варіанти — швидко. В кінці фази —
спека зберігається у `docs/specs/...-spec.md`. _Говорити:_ «контекст смертний, файли
ні: спека переживе /compact і навіть нову сесію».

**Phase 2 — DESIGN (~4–5 хв).** У звичайному режимі: ОДНЕ повідомлення з трьома
паралельними викликами design-explorer — покажи це в транскрипті. Поки агенти
працюють (це ~хвилина тиші — ідеальний момент!), відкрий
`.claude/agents/design-explorer.md` і пройдись: read-only tools (не може нічого
зламати), `model: sonnet` (дешевше), `maxTurns: 15` (стоп-кран), і ключовий рядок
інструкції — _«Do not hedge toward a middle ground — the value of parallel
exploration is in the extremes»_. _Говорити:_ «агенти не бачать один одного — тому
нема якоріння на першій ідеї; саме це втрачається в solo-режимі». Коли звіти
повернулись — три різні інтерфейси поруч. Тут запусти `/context`: історія майже не
виросла, бо в неї впали лише три короткі звіти, а не процес їх народження.

**Phase 3 — REVIEW: тут СТОП на демо.** Обери дизайн (або скажи «синтезуй 1+3»),
подивись Implementation Plan — список ПОВЕДІНОК (не кроків), кожна = один TDD-цикл.
_Говорити:_ «далі BUILD жене це через tdd-скіл: red-green-refactor по одній поведінці.
Показувати 10 хвилин написання коду не буду — але фази 4–5 працюють так само:
скіл не пустить далі, поки typecheck/тести червоні». Чесна примітка, якщо спитають:
у цьому репо тест-ранер не налаштований, тож BUILD тут спирається на typecheck;
повний TDD-цикл демонструється на проєкті з тестами.

### Уроки «що писати в скілах і для чого» (проговорити по ходу)

Це відповідь на питання слухача «окей, а ЯК мені написати такий скіл собі?» —
кожен пункт: що написано → яку поведінку це купує:

1. **`disable-model-invocation: true`** → скіл не самозапускається. Без цього
   агресивний description («Use when: implementing a new feature...») тригерив
   5-фазний процес на будь-яке «зроби фічу». Дія з великими наслідками = ручний запуск.
2. **`$ARGUMENTS` + guard на порожнє** → можна писати `/feature-forge <опис>` одним
   рядком, а випадковий Enter не ламає демо — скіл просто питає, що будуємо.
3. **Явні фази + «No proceeding without approval»** → без цього модель «пролітає»
   в код на першому ж повідомленні. Текст скіла = поведінкові рейки.
4. **Композиція скілів по імені** (grill-me → design-an-interface → tdd) → маленькі
   скіли перевикористовуються окремо; оркестратор лише тримає процес.
5. **«Якщо відповідь є в кодовій базі — дослідь код, не питай»** → один рядок, який
   прибирає 80% дурних питань в інтерв'ю.
6. **Persist артефактів у файли** (спека в docs/specs) → довгі процеси переживають
   ліміти контексту.
7. **Брифи субагентам: різні обмеження + жорсткий формат звіту + ліміт рядків** →
   паралельні агенти повертають порівнюване, а не три есеї.
8. **Секція Enforcement Rules** → правила, які скіл цитує тобі ж, коли ти намагаєшся
   зрізати кут (спробуй на демо сказати «давай одразу код» — він відмовить).
9. **TodoWrite на старті** → глядач (і ти) бачиш прогрес фаз у тудушках.
